# Wild Offscreen

Расширение для SillyTavern, генерирует автономные закадровые события для второстепенных персонажей, которые не появлялись в текущей сцене. События генерируются отдельным api запросом, хранятся по каждому персонажу и чату, и добавляются в контекст чата при каждой генерации.

В расширении всего 4 раздела:
1. Characters - содержит список персонажей и основные кнопки управления генерацией;
2. Lorebook & Context - настройка контекста и сканирования лорбука;
3. NPC Management - добавление новых персонажей и управление уже существувющими;
4. API & Language - настройки языка ивентов и профиля подключения (профиль отдельно сохраняйте в Connection Profile или используйте основной api для генераций).

## Как это работает

1. Вы сканируете лорбук (кнопкой Scan Lorebook в разделе Characters) или добавляете NPC вручную. Расширение считывает все записи и сохраняет имена, описания и поисковые ключи персонажей. Если была добавлена ненужная запись из лорбука - можно вручную удалить ее  крестиком справа от записи.
2. Каждые N сообщений бота (настраивается в Lorebook & Context) расширение вызывает отдельную модель для генерации через выбранный Connection Profile.
3. В промпт отправляются: персонаж бота ({{char}}), персона пользователя ({{user}}), информация о текущей сцене (дата, время, локация, присутствующие персонажи - либо из инфоблока (только в пресете WT) либо вручную), и блок по каждому NPC с описанием, заметками, закреплёнными фактами и историей событий.
4. Модель возвращает по одной строке на каждого NPC в формате `NPC1: локация | масштаб события | описание события`. NPC, которые находятся в активной сцене, получают статус `in-scene` - событие для них не генерируется.
5. Результаты сохраняются и добавляются в контекст основного чата в виде блока `[OFF-SCREEN NPC UPDATES]`.

## Настройка лорбука

Расширение сканирует записи лорбука в поисках карточек NPC. Запись считается карточкой NPC если выполняется **хотя бы одно** из условий:

- Поле `position` равно `0` или `"before_char"` (Char↑)/ `"after_char"` (Char↓) - отдельно настраивается в разделе Lorebook & Context (**Scan entries at position**);
- В массиве `key` (ключевые слова) есть строка `character`.

После сканирования:
- Поле `comment` (название записи) становится отображаемым именем NPC;
- Поле `content` - описанием, которое отправляется в промпт;
- Массив `key` сохраняется как поисковые ключи и используется для определения присутствия в сцене.


### Поисковые ключи и регулярные выражения

Все ключевыые слова сохраняются вместе с NPC и используются в двух местах:

- **определение присутствия в сцене** - совпадение имени NPC со списком персонажей в инфоблоке;
- **поиск упоминаний в истории чата** - нахождение сообщений, где этот NPC участвовал в сцене.

Ключи могут быть обычными строками или регулярными выражениями в формате `/паттерн/флаги`:

```
"Парфёнов"                                     ← обычное вхождение подстроки
/\bParfyonov\b/i                               ← регекс
/(?:^|[^а-яёА-ЯЁ])Савелий(?=[^а-яёА-ЯЁ]|$)/i   ← регекс с границей слова
```

Это позволяет NPC с английским именем в лорбуке корректно определяться в сцене, где его имя написано по-русски - достаточно добавить соответствующие ключи в массив `key`.

## Формат инфоблока

Расширение считывает информацию о сцене из стилизованного HTML-блока, который основная модель пишет в начале каждого сообщения (это должно быть в промпте пресета). Блок должен строго соответствовать формату.

### Обязательный формат

```html
<div style="...">
год/месяц/день • часы:минуты • Локация • Все персонажи в локации через запятую
</div>
```

Содержимое блока делится по символу `•` (U+2022):

| Сегмент | Содержимое | Пример |
|---------|-----------|--------|
| `parts[0]` | Дата в формате `ГГГГ/ММ/ДД` | `2008/11/12` |
| `parts[1]` | Время в формате `ЧЧ:ММ` | `14:15` |
| `parts[2]` | Текущая локация | `Тюменское ГУВД, кабинет Парфёнова` |
| `parts[3]` | Персонажи в сцене через запятую | `Савелий, Андрей` |

### Нормализация даты

Если основная модель пишет месяц словом (`2008/Ноябрь/12`), расширение автоматически приводит его к числовому формату `2008/11/12`. Поддерживаются русский (все падежи) и английский языки.

### Что извлекается из инфоблока

- **дата и время** → отображаются в интерфейсе и отправляются в промпт вместе с меткой времени суток (утро, день, вечер, ночь);
- **локация** → передаётся в промпт чтобы модель знала где происходит активная сцена;
- **список персонажей** → сопоставляется со всеми зарегистрированными NPC для определения кто `[IN SCENE]`, а кто `[OFFSCREEN]`.

### Логика определения присутствия в сцене

Для каждого NPC расширение проверяет совпадение с `parts[3]` в три уровня:

1. **прямое совпадение** - `"Парфёнов"` найдёт `"Савелий Парфёнов"`;
2. **обычный ключ** - любой ключ из лорбука как подстрока в списке персонажей;
3. **ключ-регекс** - любой `/паттерн/флаги` ключ тестируется против строки со всеми персонажами сцены.

NPC, определённые как присутствующие в сцене, получают статус `[IN SCENE]` и **полностью исключаются из API-запроса** - на них не тратятся токены, их `lastLocation` не обновляется.

## Режимы определения сцены

В разделе **Lorebook & Context → Scene Detection Mode** выбирается один из двух режимов.

### Info Block (по умолчанию)

Расширение читает инфоблок из последнего бот-сообщения (формат описан в разделе [Формат инфоблока](#формат-инфоблока)). Из него берутся дата, время, локация и список персонажей в сцене. Этот режим точный — NPC определяется как присутствующий только если его имя явно указано в инфоблоке.

### Text Scan

Используется когда инфоблока нет (другой пресет, или дата/локация пишутся просто текстом в сообщении). В этом режиме:

- **Определение присутствия в сцене** — расширение проверяет последние N бот-сообщений (настраивается в **Messages to scan for keywords**) на наличие имени NPC или его поисковых ключей в тексте. Если нашло — NPC считается в сцене и событие для него не генерируется.
- **Дата и время** — берутся не из инфоблока, а **вводятся вручную**. В разделе Characters появляются два поля: дата () и время (). После установки время **автоматически двигается вперёд** на заданное количество минут после каждого бот-сообщения (настраивается в **Auto +N min per exchange**, по умолчанию 5 минут). Это позволяет держать хоть какое-то ощущение хронологии без инфоблока.
- **Контекст для промпта** — при поиске сцен с NPC расширение ищет сообщения, где его имя упоминается в тексте (а не в инфоблоке).

> **Важно:** в Text Scan режиме поиск по тексту менее точный, чем инфоблок — если имя персонажа упоминается в чужой реплике или в прошедшем времени, он всё равно может быть посчитан присутствующим в сцене.

## Система событий

### Масштаб (бросок d20)

Для каждого NPC делается независимый бросок:

| Результат | Масштаб | Значение |
|-----------|---------|----------|
| 1–8 | **MINOR** | Мелкий, забываемый момент. Обычная жизнь. Ничего не меняется навсегда. |
| 9–16 | **NOTABLE** | Реальный сдвиг в жизни персонажа на этой неделе. Что-то, что он запомнит. |
| 17–20 | **MAJOR** | Меняющее жизнь событие. Необратимое или очень значительное. Здоровье, отношения, работа, идентичность. |

### Категории

Пять категорий, выбираются случайно для каждого события:

- **Personal** - внутреннее состояние, тело, привычки, личная жизнь;
- **Relationship** - реальный контакт, конфликт, откровение или изменение связи с конкретным человеком;
- **Status** - изменение материального, социального, правового или физического положения;
- **Discovery** - обнаружение чего-то нового: информации, тайны, истины;
- **Social** - участие в групповой динамике, общественной жизни, публичных ситуациях.

### Архетипы событий

Каждая комбинация масштаб + категория + тон (позитив/негатив) соответствует пулу из 20 нарративных архетипов. Расширение случайно выбирает один и отдаёт его в промпт как директиву `EVENT ARCHETYPE`.

### Формат ответа

Модель должна вернуть по одной строке на NPC в точно таком формате:

```
NPC1: локация | minor/notable/major | Описание события.
NPC2: локация | in-scene | No offscreen events. Currently in scene.
```

Расширение парсит это через `parseBatchResponse`. Если AI использует имена персонажей вместо меток `NPC1`, `NPC2` и т.д., срабатывает запасной парсер, он пробует найти совпадение по первому слову имени.

### Автопродвижение в закреплённые факты

Если модель самостоятельно указывает `major`, это событие автоматически добавляется в список `permanentFacts` NPC. Подробнее - в разделе [Закреплённые факты](#закреплённые-факты).

## Закреплённые факты

Закреплённые факты - это вещи об NPC, которые **постоянны в контексте**. Они хранятся отдельно от скользящей истории событий и инжектируются в каждый промпт как `PERMANENT FACTS (always true, never contradict these)`.

### Как добавляются

- **Автоматически** - любое событие, которое AI сам пометил как `major`, автоматически становится фактом;
- **Вручную** - кнопка 📌 рядом с любым событием в интерфейсе закрепляет его как постоянный факт.

### Как удаляются

Кнопка ✕ рядом с фактом в интерфейсе удаляет его.

### Область видимости

Закреплённые факты хранятся отдельно для каждого бота и каждого чата. При создании нового чата с тем же персонажем факты очищаются вместе с событиями (см. [Структура хранилища](#структура-хранилища)).

### Дата истории

При создании факта (авто или вручную) расширение сохраняет внутриигровые дату и время из текущего инфоблока (например `2008/11/12 14:15`). Это сохраняется вместе с фактом и отображается в интерфейсе для ориентировки на таймлайне.

## Управление NPC

### Ручное добавление NPC

NPC можно добавить без записи в лорбуке через раздел **NPC Management → Add new NPC**. Укажите имя и опциональное описание. Добавленные вручную NPC получают флаг `pendingIntro: true` (см. [Ввод нового NPC в сцену](#ввод-нового-npc-в-сцену)).

### Редактирование NPC

Раздел **NPC Management → Edit existing NPC** предоставляет:

- **Notes** (Заметки) - свободный текст, который всегда отправляется в промпт с пометкой `CURRENT NOTES (override lorebook if contradicts)`. Используйте для актуального состояния персонажа: `сейчас беременна`, `в конфликте с Парфёновым`. Заметки хранятся для всех чатов с этим ботом;
- **Description** (Описание) - редактируемая версия описания из лорбука. Оригинальный текст лорбука сохраняется отдельно; кнопка **Reset desc** восстанавливает его.

### Отключение NPC

Кнопка паузы на карточке NPC отключает генерацию событий для этого персонажа без его удаления. Отключённые NPC исключаются из API-запросов.

### Очистка и удаление

- **Корзина на карточке** - удаляет все события и закреплённые факты этого NPC в текущем чате;
- **✕ на карточке** - полностью удаляет NPC из всех чатов;
- **Clear All Events** - очищает историю событий всех NPC в текущем чате (факты сохраняются);
- **Remove All** - удаляет всех NPC полностью.

### Редактирование текста события

Текст события в карточке NPC можно редактировать прямо на месте., кликнув по тексту. Изменения сохраняются мгновенно без перерендера карточки. Escape отменяет изменения и восстанавливает оригинал.

## Ввод нового NPC в сцену

При добавлении NPC вручную он получает флаг `pendingIntro: true`. Это влияет на инжекцию двумя способами:

### Пассивный режим

В контекст добавляется тихая строка:
```
[PENDING: Имя - exists in this world, introduce naturally when the moment fits.]
```
Основная модель знает что персонаж существует, но не обязана вводить его прямо сейчас.

### Активный режим (кнопка введения)

Кнопка с иконкой двери на карточке NPC ставит в очередь активное введение. При следующей генерации в конец инжекта добавляется директива:

```
[SCENE DIRECTION — REQUIRED]
Introduce the following character(s) into this response: Имя.
They must appear or be directly referenced in this message — not hinted at, not delayed.
The introduction must feel natural and fit the current location, time, and tone.

Character: Имя
...описание персонажа из лорбука...
[END SCENE DIRECTION]
```

Описание персонажа из лорбука добавляется прямо в директиву — чтобы модель знала кого вводить, даже если карточка NPC не попала в контекст другим путём. Это срабатывает один раз и автоматически сбрасывается после следующего ответа бота, независимо от того появился ли персонаж в тексте.

### Автоопределение

После каждого ответа бота расширение проверяет присутствие имени NPC в списке персонажей инфоблока. Если совпадение найдено - флаг `pendingIntro` снимается автоматически, показывается уведомление `"Имя has entered the scene!"`.

## Структура хранилища

Данные хранятся в `extension_settings` SillyTavern под ключом `wild-offscreen`:

```
npcData
└── [botKey]                       ← по аватару или имени персонажа-бота
    ├── __npcs                     ← данные идентичности (общие для всех чатов)
    │   └── [имя]
    │       ├── name
    │       ├── description        ← текущее описание (может быть отредактировано)
    │       ├── lorebookDescription ← оригинал из лорбука (для Reset)
    │       ├── notes              ← поле заметок
    │       ├── enabled
    │       ├── searchKeys
    │       ├── lorebookName
    │       ├── entryUid
    │       └── pendingIntro
    └── [chatKey]                  ← данные времени выполнения (отдельно на каждый чат)
        └── [имя]
            ├── events[]           ← скользящая история событий (макс. N штук)
            │   └── { text, location, scale, category, positive, timestamp, storyDate }
            ├── permanentFacts[]
            │   └── { text, category, positive, timestamp, storyDate, auto }
            └── lastLocation
```

### Новый чат vs возврат к старому

При смене чата расширение сравнивает ключ бота и ключ чата. Если ключ чата новый и данных по нему нет - все события и факты очищаются, показывается уведомление. **Данные идентичности NPC (имена, описания) сохраняются.** 

При возврате к старому чату с тем же персонажем все события и факты этого чата восстанавливаются автоматически.

## Справочник настроек

| Настройка | По умолчанию | Описание |
|-----------|-------------|----------|
| `enabled` | `true` | Главный переключатель вкл/выкл |
| `triggerEvery` | `1` | Генерировать события каждые N ответов бота (1 = каждый ответ, 2 = каждый второй и т.д.) |
| `maxEvents` | `7` | Максимальное количество хранимых событий на NPC в одном чате |
| `maxMessages` | `30` | Сколько последних сообщений читать для контекста чата |
| `maxCharsPerMsg` | `2000` | Ограничение символов на сообщение при построении контекста |
| `maxTokens` | `200` | Базовое значение max_tokens (реальное: `max(400, количество_NPC × 120)`) |
| `connectionProfile` | `''` | Имя Connection Profile для API-вызовов |
| `outputLanguage` | `'en'` | Язык генерируемых событий: `en`, `ru` или `uk` |
| `scanPosition` | `'before_char'` | Позиция записей лорбука для сканирования: `before_char` или `after_char` |

## Разделы интерфейса

Панель расширения состоит из четырёх раскрывающихся секций:

### Characters (открыта по умолчанию)
- Текущие дата и время из последнего инфоблока
- Список карточек NPC с событиями и закреплёнными фактами
- **Generate Now** - немедленный запуск цикла генерации
- **Scan Lorebook** - сканирование прикреплённых лорбуков
- **Clear All Events** - очистка истории событий всех NPC в текущем чате (факты сохраняются)
- **Remove All** - полное удаление всех NPC

### Lorebook & Context
- Имя лорбука и выбор позиции сканирования
- Глубина истории сообщений
- Максимум символов на сообщение
- Частота генерации событий и максимум хранимых событий

### NPC Management
- Добавление нового NPC вручную (имя + описание)
- Редактирование существующего NPC: выбор из списка, редактирование заметок и описания, сброс описания к оригиналу из лорбука

### API & Language
- Выбор языка вывода (EN / RU / UK)
- Выбор Connection Profile с кнопкой обновления

## Нюансы и поведение

**Определение реролла** - При реролле расширение замечает что длина чата не изменилась, но содержимое последнего бот-сообщения изменилось (сравнивается по фингерпринту из последних 60 символов). В этом случае последнее событие у каждого NPC удаляется, и сразу запускается новая генерация - таким образом события всегда соответствуют актуальному варианту сцены. Закреплённые факты (MAJOR и вручную закреплённые) при реролле **не затрагиваются никогда**.

**Новый чат vs возврат к старому** - Расширение отслеживает ID чатов. Возврат к предыдущему чату с тем же персонажем восстанавливает все события и факты из той сессии.

**NPC в сцене полностью исключаются** - Они фильтруются до отправки API-запроса. Никаких токенов на них не тратится, их `lastLocation` не меняется.

**Запасной парсер** - Если AI отвечает `Илья: школьный коридор | minor | ...` вместо `NPC1: ...`, парсер пробует найти совпадение по первому слову имени. В консоли появится запись `Fallback name match for NPC1 (Илья)`.

**Счётчик сообщений и триггер** - Расширение считает **только ответы бота** (события `CHARACTER_MESSAGE_RENDERED` и `MESSAGE_RECEIVED`). Стартовое сообщение бота в начале чата игнорируется - счётчик начинает считать с первого ответа после того как пользователь написал. При `triggerEvery = 1` генерация запускается на каждый ответ бота; при `triggerEvery = 2` - через ответ. Дедупликация по фингерпринту сообщения (индекс + последние 60 символов) гарантирует ровно один запуск на сообщение, даже если оба события сработали почти одновременно.

**Язык вывода** - Настройка языка добавляет одну строку в системный промпт (`Write in Russian.` и т.д.) и влияет только на текст событий, но не на служебные метки вроде `NPC1:` или слов масштаба.

**Счётчик токенов** - Отображение `~N tokens injected` в интерфейсе - приблизительная оценка на основе длины текста, делённой на 4 (подходит для смешанного русского/английского текста).

**Заметки vs Описание** - Описание - полный бэкграунд персонажа для AI. Заметки - краткое актуальное состояние с более высоким приоритетом. Используйте заметки для вещей, меняющихся по ходу истории: `сейчас под следствием`, `скрывает что-то от Парфёнова`.

**Поисковые ключи и определение сцены** - Если имя персонажа в лорбуке написано по-английски, а в инфоблоке по-русски - добавьте русские формы имени (обычные или regex) в массив `key` записи лорбука. Расширение будет использовать их для определения присутствия в сцене.
